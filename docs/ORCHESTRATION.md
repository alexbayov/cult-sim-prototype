# Orchestration

Этот проект можно вести в двух режимах:

1. **Один Devin** — быстрые последовательные PR.
2. **Дирижёр + девы** — параллельная работа без потери связности.

Второй режим нужен, когда задача делится на 2–3 независимых трека.

Для полного контекста следующей сессии см. `docs/CONDUCTOR_HANDOFF.md`.

## Когда включать orchestration mode

Использовать, если:

- есть 2–3 независимых направления;
- один PR не должен менять те же файлы, что другой;
- нужно экономить контекст основной сессии;
- человек-оркестратор готов мержить и запускать smoke tests.

Не использовать, если:

- задача маленькая;
- рефакторинг монолитный;
- все изменения конфликтуют в одних и тех же файлах;
- нет ясных acceptance criteria.

## Роли

### Дирижёр

Держит общую картину.

Делает:

- читает roadmap/backlog;
- декомпозирует работу;
- пишет brief для девов;
- ревьюит PR;
- даёт smoke-команды;
- предлагает следующий раунд.

Не делает:

- не пишет код параллельно с девами;
- не мержит PR;
- не меняет границы задачи без согласования.

### Дев

Работает по одному brief.

Делает:

- меняет только указанные области;
- открывает PR;
- пишет тесты/проверки в PR;
- фиксит замечания.

Не делает:

- не лезет в чужой трек;
- не мержит сам;
- не расширяет scope.

### Человек-оркестратор

Делает:

- запускает сессии;
- передаёт brief девам;
- мержит PR;
- запускает smoke-команды;
- принимает финальные решения.

## Хорошие параллельные треки для этого проекта

### Раунд A — playable foundation

Можно разделить:

1. Save/load + platform storage adapter.
2. JSON content migration + validation.
3. Mobile-first layout.

### Раунд B — content expansion

Можно разделить:

1. Карты инфоцыганского сценария.
2. Участники и финалы.
3. Комбо и баланс.

### Раунд C — visual production

Можно разделить:

1. Портреты участников.
2. Карточные иллюстрации.
3. Иконки/фон/UI polish.

## Brief template

```md
# Dev N — Task Brief: <короткое название>

Repo: https://github.com/alexbayov/cult-sim-prototype
Branch: devin/<task-name>
Coordinator: conductor Devin
Reviewer: conductor Devin + human orchestrator

## Контекст

Почему эта задача важна и как она связана с roadmap.

## Что менять

- Файл/папка 1: что именно сделать.
- Файл/папка 2: что именно сделать.

## Что НЕ делать

- Не менять чужой трек.
- Не расширять scope.
- Не делать большой refactor без необходимости.

## Acceptance criteria

- [ ] Критерий 1.
- [ ] Критерий 2.
- [ ] `npm run build` проходит.
- [ ] `npm run lint` проходит.

## PR-чеклист

- [ ] Summary написан.
- [ ] Тесты/проверки указаны.
- [ ] Открытые вопросы указаны.

## Smoke commands

```bash
npm install
npm run build
npm run lint
npm run dev
```
```

## Smoke tests

Базовые команды:

```bash
npm install
npm run build
npm run lint
npm run dev
```

Ручная проверка для текущей игры:

1. Открыть приложение.
2. Сделать 2–3 хода.
3. Проверить изменение метрик.
4. Проверить журнал.
5. Обновить страницу.
6. Проверить сохранение состояния.
7. Нажать «новая партия».
8. Проверить сброс.

## Анти-паттерны

- Три агента меняют `src/App.tsx` одновременно.
- Нет brief, только «сделай красиво».
- Нет smoke-команд.
- Дев добавляет новый сценарий до стабилизации первого loop.
- Визуальные ассеты подключаются без `asset_manifest.csv`.

## Постоянное правило

Дирижёр всегда должен держать `docs/ROADMAP.md`, `docs/BACKLOG.md`, `docs/AGENT_BRIEF.md` и этот файл актуальными.
